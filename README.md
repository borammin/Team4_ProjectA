## Project Overview

This project implements an event-driven data pipeline that automatically converts uploaded CSV files into JSON format using AWS services. When a `.csv` file is uploaded to the input S3 bucket, it triggers a Lambda function that converts it to JSON, stores the output in a separate output S3 bucket, and logs the metadata to a relational database.

---

## Architecture

| Layer | Service |
|-------|---------|
| Storage | AWS S3 |
| Compute | AWS Lambda (event-driven) |
| Database | AWS RDS MySQL |

## Installation & Deployment

### 1. Storage Configuration

Create two S3 buckets in `us-east-1`:
- `ds2002-team4-input` — receives uploaded CSV files
- `ds2002-team4-output` — stores converted JSON files

Keeping buckets separate prevents accidental re-triggering of the pipeline and avoids infinite execution loops.

### 2. Database Initialization

Connect to your RDS MySQL instance and run the following in the `team4` database:

```sql
CREATE TABLE IF NOT EXISTS processing_logs (
    id INT AUTO_INCREMENT PRIMARY KEY,
    input_key VARCHAR(255),
    output_key VARCHAR(255),
    bucket_name VARCHAR(100),
    timestamp DATETIME,
    status VARCHAR(20),
    error_message TEXT
);
```

> **Assumption:** The RDS instance is already running and accessible from within the same AWS VPC as the Lambda function.

### 3. Lambda Function Setup

**Function name:** `csv-to-json-converter`  
**Runtime:** Python 3.12  

#### IAM Execution Role
Attach the following policies to the Lambda execution role:
- `AmazonS3FullAccess` (or scoped: `s3:GetObject` on input bucket, `s3:PutObject` on output bucket)
- `AWSLambdaBasicExecutionRole` (enables CloudWatch logging)

#### pymysql Dependency (Lambda Layer)
`pymysql` is not built into the Lambda runtime. Package it as a layer:

```bash
mkdir pymysql-layer && cd pymysql-layer
mkdir python
pip install pymysql -t python/
zip -r pymysql-layer.zip python/
```

Then in AWS Console: **Lambda → Layers → Create layer** → upload `pymysql-layer.zip` → attach to `csv-to-json-converter`.

#### Environment Variables

Set the following in **Lambda → Configuration → Environment variables**:

| Key | Value |
|-----|-------|
| `DB_HOST` | Your RDS endpoint |
| `DB_USER` | Your database username |
| `DB_PASS` | Your database password |
| `DB_NAME` | `team4` |
| `OUTPUT_BUCKET` | `ds2002-team4-output` |

> **Note:** AWS credentials are NOT hardcoded. The Lambda uses its IAM execution role automatically via boto3.

#### S3 Trigger
Add an S3 Event Notification on `ds2002-team4-input`:
- **Event type:** All object create events
- **Suffix filter:** `.csv`
- **Destination:** Lambda function `csv-to-json-converter`

## Pipeline Logic

1. **Extract:** Lambda is triggered by an S3 `ObjectCreated` event. It captures the bucket name and object key, then downloads the CSV into memory using `boto3`.
2. **Transform:** Uses Python's `csv.DictReader` to parse CSV rows into a list of dictionaries, then serializes to JSON using `json.dumps`.
3. **Load:**
   - Uploads the JSON file to `ds2002-team4-output` with the same filename (`.csv` → `.json`)
   - Logs the result to the RDS `processing_logs` table via `pymysql`, including timestamp and status

## Usage

### Upload a CSV file
```bash
aws s3 cp my_data.csv s3://ds2002-team4-input
```

The pipeline will automatically trigger, convert the file, and store the JSON output in `ds2002-team4-output`.

### Query processed files
```bash
python3 query.py my_data.csv
```

---

## Key Design Choices

- **Event-driven architecture:** Lambda only runs when a file is uploaded — no idle compute costs
- **Separate input/output buckets:** Prevents the output JSON from re-triggering the pipeline
- **Relational logging:** Every conversion is logged with its status, enabling easy auditing and debugging
- **No hardcoded credentials:** boto3 uses the Lambda execution role automatically
